## 🗺️ タスク管理フロー

### Issue作成からクローズまで

```
1. Issue作成
   - テンプレート選択（Feature/Bug/Refactor）
   - タイトルと説明を記入
   - ラベル、マイルストーン、担当者を設定
   ↓
2. 開発開始
   - ブランチ作成（feature/MS1-xxx）
   - Issue番号を含むコミット
   ↓
3. PR作成
   - Issue番号を参照
   - レビュー依頼
   ↓
4. マージ
   - Squash and Merge
   - Issue自動クローズ
```

## 📊 タスク優先度

### 優先度ラベル

| ラベル         | 説明                       | 対応期限  |
| -------------- | -------------------------- | --------- |
| 🔴 **Critical** | システム停止等の重大な問題 | 即日対応  |
| 🟠 **High**     | MVP必須機能、重要なバグ    | 3日以内   |
| 🟡 **Medium**   | 重要だが緊急ではない機能   | 1週間以内 |
| 🟢 **Low**      | 将来的な改善項目           | 2週間以降 |

## 🚀 クイックスタート

### 新しいIssueを作成する

```markdown
1. GitHubのIssuesタブに移動
2. "New Issue"ボタンをクリック
3. テンプレートを選択:
   - Feature Request (新機能)
   - Bug Report (バグ報告)
   - Refactoring (リファクタリング)
4. 必要事項を記入:
   - タイトル
   - 説明
   - 受け入れ基準
5. ラベル、マイルストーン、担当者を設定
6. "Submit new issue"で作成
```

### マイルストーンの進捗確認

```markdown
1. GitHubのMilestonesタブに移動
2. 該当するマイルストーンをクリック
3. 進捗率と完了/未完了のIssueを確認
```

## 📊 タスク管理のベストプラクティス

### Issue作成時
- **明確なタイトル**: 何をするかが一目でわかるタイトル
- **詳細な説明**: 背景、目的、実装方法を記載
- **受け入れ基準**: 完了条件を明確にリスト化
- **適切なラベル**: 種類、優先度、マイルストーンを設定

### Issue管理時
- **定期的な確認**: 週1回は全Issueをレビュー
- **優先度の調整**: 状況に応じて柔軟に変更
- **古いIssueの整理**: 3ヶ月以上放置されたIssueを見直し
- **マイルストーンの更新**: 進捗に応じて調整

### コミュニケーション
- **Issue内でディスカッション**: 決定事項をIssueコメントに記録
- **進捗報告**: 週次で進捗をコメント
- **ブロッカーの報告**: 問題があれば早めに共有

## ❓ よくある質問

### Q: Issueとタスクの違いは？

**A:** GitHubではIssueがタスク管理の単位です。1つのIssue = 1つのタスクとして扱います。

### Q: Issue番号はどう割り振られますか？

**A:** GitHubが自動的に連番で割り振ります。Issue #1, #2, #3...のように管理されます。

### Q: マイルストーンはいつ作成しますか？

**A:** 大きな機能追加やリリース計画時に作成します。詳細は [Milestones](./milestones.md) を参照してください。

### Q: Issueが増えすぎて管理できません

**A:** 以下の方法で整理してください:
- ラベルでフィルタリング
- マイルストーンでグループ化
- 古いIssueをクローズ（必要なら再オープン）
- プロジェクトボードで可視化
